🛡️ 1. 安全红线（绝对禁区）
违反即可能导致数据泄露或合规风险，必须零容忍
禁止数据脱敏失败
  • ❌ 禁止直接粘贴：AK/SK、数据库密码、客户 PII（个人敏感信息）
  • ❌ 禁止上传真实数据：生产环境配置文件、客户订单数据、日志明文
  • ✅ 必须先脱敏：用 ****** 或占位符替换后再发送
禁止核心业务逻辑泄露
  • ❌ 禁止全量上传：公司核心商业逻辑（定价算法、风控规则）到公有云 AI
  • ✅ 仅上传抽象代码：通用技术逻辑、经过抽象的伪代码
禁止开源协议污染
  • ❌ 不审查直接用：AI 生成的复杂算法可能来自 GPL 等传染性协议
  • ✅ 必须复核：使用代码扫描工具或人工审查开源协议风险
💻 2. 核心理念（责任归属）
AI 是副驾驶，你是主驾驶，代码最终由你负责
原则 1你是代码的主人
❌ 错误心态："这是 AI 写的，bug 不怪我"
✅ 正确心态：AI 只是辅助工具，提交的代码必须由你完全理解并负责
💡 行动：任何 AI 引入的 Bug 都是你的 Bug，必须人工 Review 后再提交
原则 2保持批判性思维
❌ 盲目信任：不要盲目信任首次生成的代码
✅ 适度怀疑：AI 可能产生幻觉（最新日期、特定库版本、边界条件）
💡 行动：必须进行人工 Review，特别是涉及安全、性能、边界的代码
原则 3粒度控制
❌ 避免：一次性让 AI 生成整个模块或超长文件
✅ 推荐：将大需求拆解为接口定义、核心逻辑、单元测试三步走
💡 效果：保持对代码的掌控力，便于 Review 和调试
原则 4测试驱动
✅ 强制要求：生成代码必生成对应的单元测试
💡 效果：通过测试用例反向验证 AI 生成的业务逻辑是否正确
🚀 3. 快速上手（3 步启动）
新人必读：从零开始使用 AI 辅助编程的最小化路径
Step 1环境准备（10分钟）
  • ✅ 安装 AI 工具：通义灵码 / GitHub Copilot / Cursor（选其一）
  • ✅ 配置企业知识库：登录企业账号，确认部门规范已同步
  • ✅ 学习快捷键：Ctrl+I 唤起对话，Tab 接受建议
Step 2试跑场景（30分钟）
推荐新人第一个任务：生成单元测试（容错率高，易于验证）
示例提示词：
根据公司单测规范，为以下方法生成单元测试：
[粘贴方法代码]
要求：覆盖正常场景、边界条件、异常情况
Step 3逐步扩展（1-2周）
  • Week 1：使用 AI 生成样板代码（CRUD、接口定义）
  • Week 2：使用 AI 辅助代码重构、生成文档注释
  • 持续：收藏好用的提示词模板，沉淀到个人知识库
💬 4. 如何与 AI 高效沟通（提示词工程）
好的提示词 = 清晰的上下文 + 明确的约束 + 预期的输出格式
4.1 提示词模板结构（4 要素）
要素 说明 示例
1. 角色定位 告诉 AI 扮演什么角色（可选但推荐） "你是本公司的后端工程师"
2. 背景上下文 提供技术栈、依赖库、数据结构定义 "项目使用 Spring Boot 2.7 + MyBatis-Plus"
3. 任务目标 明确要做什么（核心需求） "生成用户注册接口的 Controller 层代码"
4. 约束条件 必须遵守的规范、禁止项、输出格式 "必须使用公司的 Result 包装器，禁止硬编码错误码"
4.2 反例与正例对比（学会提问的艺术）
❌ 模糊提示词（低效） ✅ 清晰提示词（高效）
"帮我写个登录" "作为本公司后端工程师，使用 Spring Security + JWT，生成用户登录接口，入参包含 username/password，返回 JWT Token，需集成公司统一鉴权中心"
"这段代码有什么问题" "请作为代码审查专家，按公司 Java 规范审查此代码，重点检查：1. 是否存在 NullPointerException 风险；2. 事务边界是否合理；3. 变量命名是否符合规范"
"优化性能" "此方法在高并发场景下响应慢，请分析时间复杂度，找出 N+1 查询问题，并给出使用 MyBatis 批量查询的优化方案"
4.3 高频场景提示词速查（收藏此表）
场景 提示词模板
生成单测 根据公司单测规范，为以下代码生成单元测试，覆盖正常/边界/异常场景：[粘贴代码]
代码评审 作为 Tech Lead，按公司评审 Checklist 审查此代码，关注并发安全、事务粒度、规范合规
遗留代码理解 用 Mermaid 流程图画出此代码的核心执行路径，并解释业务逻辑，指出潜在风险
报错排查 我的程序报错，环境是 [技术栈]，堆栈信息：[粘贴]。请分析直接原因、关联推测、给出解决方案
重构建议 此代码可维护性差，请在不破坏逻辑的前提下，用 [设计模式/新语法] 重构，并解释改进点
⚠️ 5. 常见问题与解决方案（避坑指南）
95% 的新人会踩的坑，提前规避可节省 80% 的调试时间
❓ 问题 1：AI 生成的代码跑不通
症状：编译报错、运行时异常、API 不存在
根因：
· AI 对项目依赖版本不了解（如用了 Spring Boot 3 语法但项目是 2.x）
· AI 产生幻觉，编造了不存在的 API
解决方案：
✅ 提示词中明确依赖版本："使用 MyBatis-Plus 3.5.3.1 版本"
✅ 优先使用项目已有依赖：先查看 pom.xml 再提问
✅ 分批生成：先生成接口定义，验证通过后再生成实现
❓ 问题 2：生成的代码风格不符合团队规范
症状：变量命名混乱、缺少注释、不符合 Lint 规则
解决方案：
✅ 在提示词中引用规范："按阿里巴巴 Java 开发手册生成代码"
✅ 配置 System Prompt：将团队规范写入 .cursorrules 或通义灵码企业知识库
✅ 后置重构：生成后手动调整，或让 AI 二次优化："请按 Google Java Style 格式化"
❓ 问题 3：AI 理解错我的需求，生成的代码偏离目标
症状：生成了错误的业务逻辑、实现方式不符合预期
解决方案：
✅ 提供具体示例：给出输入输出示例、伪代码
✅ 分步引导：先让 AI 描述理解的需求，确认无误后再生成代码
✅ 追加约束："不要使用递归，用循环实现"
❓ 问题 4：每次生成的代码都不一样（不稳定）
症状：多次生成结果不一致，难以复现
解决方案：
✅ 强调过程逻辑：要求 AI 先输出处理逻辑和推导过程
✅ 固化好的版本：生成满意的代码后，将提示词和结果保存到知识库
✅ 使用确定性约束："按此伪代码严格翻译为 Java"
❓ 问题 5：AI 生成了过度设计的代码（代码膨胀）
症状：引入不必要的抽象层、工厂模式、过多接口
解决方案：
✅ 明确 KISS 原则："用最简单的方式实现，避免过度设计"
✅ Review 阶段删减：评审时删除不必要的抽象
✅ 设置复杂度上限："实现控制在 50 行以内"
📈 6. 如何持续成长（从新手到专家）
AI Coding 也有能力曲线，持续实践才能提升效能
6.1 成长路径（L1 → L4）
级别 能力特征 典型任务 提升建议
L1 初级 能使用 AI 完成简单任务，需要频繁调试 生成单测、CRUD 代码、代码格式化 多练习，收藏提示词模板
L2 中级 能独立设计提示词，解决复杂问题 接口设计、重构、性能优化、bug 排查 学习提示词工程，沉淀场景库
L3 高级 能优化团队规范，培训他人，输出最佳实践 制定团队 Prompt 库、Code Review、架构设计 对外分享，参与公司规范建设
L4 专家 行业影响力，对外输出，技术创新 制定企业级规范、技术布道、开源贡献 撰写白皮书、行业演讲
6.2 刻意练习（每周任务清单）
  • ✅ Week 1-2：每天至少用 AI 完成 1 个任务（单测、注释、格式化）
  • ✅ Week 3-4：尝试让 AI 辅助重构 1 个遗留代码模块
  • ✅ Month 2：沉淀 10 个好用的提示词模板，分享到团队知识库
  • ✅ Month 3：参与 1 次团队 AI Coding 经验分享会
  • ✅ Quarter：对比 AI 辅助前后的效率提升数据（开发周期、Bug 率）
6.3 知识沉淀（个人资产化）
建立个人 Prompt 库（推荐工具：Notion / 飞书文档 / GitHub Gist）
分类管理：
· 按场景：代码生成 / 代码评审 / 重构 / 调试 / 文档
· 按技术栈：Java / Python / JavaScript / SQL
· 按质量：⭐⭐⭐ 高频好用 / ⭐⭐ 偶尔使用 / ⭐ 存档备用
持续优化：
· 每次使用后记录效果（好用/一般/不好用）
· 定期回顾，淘汰低效模板，精简高效模板
6.4 避免倦怠期（如何坚持）
症状：初期热情高涨，1-2 个月后逐渐回到传统写法
应对策略：
✅ 设置小目标：每周至少用 AI 完成 3 个任务（具体可量化）
✅ 寻找成就感：记录效率提升数据，可视化成长曲线
✅ 团队互助：加入公司 AI Coding 兴趣小组，互相激励
✅ 奖励机制：达成里程碑时奖励自己（如：连续 30 天使用 → 买心仪设备）
✅ 正视阵痛：初期可能效率下降（J 型曲线），给自己 1-2 周缓冲期
✅ 7. 代码提交前自查清单（质量保障）
每次使用 AI 生成代码后，必须完成以下检查项才能提交
🔍 Phase 1：功能正确性（5分钟）
  • □ 本地运行通过：代码编译成功，单元测试通过
  • □ 逻辑验证：手动测试核心路径，确认业务逻辑正确
  • □ 边界测试：测试空值、极大值、并发等边界情况
  • □ 幂等性检查：重复请求是否安全（特别是 POST/PUT 接口）
🔒 Phase 2：安全合规性（3分钟）
  • □ 敏感信息：代码中无硬编码密码、AK/SK
  • □ SQL 注入：使用参数化查询，未拼接 SQL
  • □ XSS 防护：前端输出已转义，后端校验输入
  • □ 权限校验：关键操作已集成鉴权逻辑
⚡ Phase 3：性能与稳定性（3分钟）
  • □ 时间复杂度：无明显的 O(n²) 或更高复杂度循环
  • □ N+1 查询：数据库查询已优化（批量查询/JOIN）
  • □ 资源释放：I/O 流、数据库连接已关闭（try-with-resources）
  • □ 并发安全：共享变量已加锁或使用线程安全类
📐 Phase 4：规范与可维护性（3分钟）
  • □ 命名规范：变量/方法命名清晰，符合团队规范
  • □ 注释完整：关键逻辑有注释，复杂算法有说明
  • □ 魔术数字：硬编码的数字已提取为常量
  • □ 代码长度：单个方法不超过 50 行（复杂逻辑已拆分）
🧪 Phase 5：测试覆盖（2分钟）
  • □ 单元测试：核心方法已编写单元测试
  • □ 覆盖率：测试覆盖率 ≥ 80%（公司标准）
  • □ 异常场景：测试用例覆盖异常分支
💡 提交前最后确认
扪心自问：如果这段代码出了 Bug，我能在 5 分钟内定位问题吗？
如果答案是"不确定"，请继续 Review；如果是"可以"，放心提交！
