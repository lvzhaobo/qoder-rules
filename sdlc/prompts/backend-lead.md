# 后端组长提示词 (Backend Team Lead)

## 角色定位
**后端组长** - 负责代码审查、架构决策和技术债务管理

---

## 场景 1: 代码审查 (Code Review) & 架构合规

### 目标
审查代码的安全性、性能及规范性

### 提示词

```
你是一位 Tech Lead。请审查上述"秒杀接口"的伪代码，关注以下风险点：
1. **并发安全**：是否存在死锁或竞态条件？
2. **事务粒度**：数据库事务范围是否过大？是否包含了 RPC 调用？
3. **性能瓶颈**：指出代码中可能导致 QPS 上不去的具体行数。
4. **规范合规**：变量命名是否符合 [阿里巴巴 Java 开发手册]？

请输出详细的 CR 报告和重构后的代码片段。
```

### 预计输出
- 具体的代码风险点清单
- 事务与并发优化建议
- 重构后的高质量代码片段

### 效果
✅ 保证代码库的长期健康，防止技术债务累积。

---

## 场景 2: 技术债务评估 & 架构决策

### 目标
评估重构的必要性与方案

### 提示词

```
我们的老旧单体应用 [CRM系统] 面临维护困难。团队正在讨论是否迁移到微服务。
请作为首席架构师评估：
1. **现状分析**：分析单体架构在 50 人团队协作下的痛点。
2. **拆分策略**：建议按 DDD（领域驱动设计）的哪个限界上下文进行拆分？
3. **风险预警**：分布式事务（Seata/Saga）带来的复杂度成本。
4. **决策建议**：立即重构还是渐进式绞杀（Strangler Fig Pattern）？
```

### 预计输出
- 架构迁移路线图
- 微服务引入的风险评估
- 专业的架构决策建议

### 效果
✅ 避免盲目跟风技术架构，确保架构演进符合业务发展阶段。

---

## 使用建议

💡 **适用场景：**
- 日常代码 Review
- 重要代码合并前的审查
- 技术债务评估
- 架构重构决策

**注意事项：**
- Code Review 要关注核心风险点，不要纠结于代码格式细节
- 事务边界要清晰，避免在事务中执行耗时操作
- 性能瓶颈要结合压测数据，不要过度优化
- 架构决策要权衡利弊，不要盲目追求新技术
- 技术债务要定期清理，避免积重难返
