# 研发总监提示词 (R&D Director)

## 角色定位
**研发总监** - 负责系统稳定性审计和研发效能管理

---

## 场景 1: 稳定性审计 (Reliability)

### 目标
评估系统的稳定性指标

### 提示词

```
请作为 SRE 专家，根据以下 Prometheus/Grafana 监控数据快照（或日志片段），分析系统健康状况：
1. **黄金指标**：分析延迟 (Latency)、流量 (Traffic)、错误 (Errors)、饱和度 (Saturation)。
2. **异常模式**：是否存在周期性的 Full GC？是否存在连接池泄露迹象？
3. **SLO 风险**：当前错误率是否消耗了过多的错误预算 (Error Budget)？
4. **建议**：给出 3 条提升系统可用性的架构建议（如：熔断、降级、扩容）。

数据片段：`[粘贴监控数据]`
```

### 预计输出
- 系统健康度诊断报告
- SRE 视角的优化建议

### 效果
✅ 量化评估系统稳定性，用数据指导架构优化方向。

---

## 场景 2: 效能扫描 (Efficiency)

### 目标
诊断团队研发效能瓶颈

### 提示词

```
请分析团队最近一个月的 Git 提交记录和 Jira 流转记录（见附录）。
请评估：
1. **流动效率**：平均 PR 合并时间 (Cycle Time) 是多少？是否存在长期不合并的分支？
2. **质量指标**：Bug/Story 比率如何？返工率（Re-open）高吗？
3. **瓶颈识别**：是否卡在"Code Review"环节？还是卡在"测试"环节？
4. **改进计划**：针对识别出的瓶颈，提出 DORA 指标提升计划。

数据摘要：`[粘贴统计数据]`
```

### 预计输出
- 研发效能诊断报告
- DORA 指标分析
- 流程改进建议

### 效果
✅ 识别研发流程中的"阻滞点"，提升团队交付吞吐量。

---

## 使用建议

💡 **适用场景：**
- 系统稳定性定期审计
- 研发效能季度/月度评估
- 团队流程优化决策
- 向上汇报的数据支撑

**注意事项：**
- 稳定性审计应建立定期机制，不要等出问题才看
- DORA 指标（部署频率、变更前置时间、MTTR、变更失败率）是衡量研发效能的关键
- 关注瓶颈环节，不要平均用力
- 数据驱动决策，避免凭感觉做判断
