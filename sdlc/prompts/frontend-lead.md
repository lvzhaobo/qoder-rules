# 前端组长提示词 (Frontend Team Lead)

## 角色定位
**前端组长** - 负责性能审计、工程化规范和前后端协作

---

## 场景 1: 性能审计 & 工程化规范

### 目标
审查组件的性能、可维护性及 Hook 使用

### 提示词

```
请 Review 上述 `UserSelector` 组件：
1. **渲染性能**：`useMemo` 和 `useCallback` 的使用是否正确？是否存在不必要的重渲染？
2. **状态管理**：是否应该将搜索逻辑抽离为 `useUserSearch` Hook？
3. **Prop Drilling**：Props 定义是否过于复杂？
4. **可访问性**：是否支持键盘操作（A11y）？

请指出代码中的"坏味道"并提供优化方案。
```

### 预计输出
- 性能瓶颈分析
- 代码坏味道清单
- 架构优化建议

### 效果
✅ 防止低质量代码进入主分支，保证前端应用的流畅度。

---

## 场景 2: 跨团队联调 (Contract)

### 目标
生成 Mock 数据以解除对后端的依赖

### 提示词

```
后端接口 `GET /api/users` 还没开发完，但我需要先开发前端 UI。
请根据这份 Swagger/OpenAPI 定义（见下文）生成 Mock 数据：
1. **工具**：使用 [Mock.js / MSW]。
2. **场景**：生成 20 条数据，包含正常用户、封禁用户、超长名字用户。
3. **延迟**：模拟 500ms 网络延迟。

OpenAPI 定义：`[粘贴 YAML/JSON]`
```

### 预计输出
- 开箱即用的 Mock 脚本
- 覆盖边界情况的测试数据

### 效果
✅ 前后端并行开发，互不阻塞，提升整体交付效率。

---

## 使用建议

💡 **适用场景：**
- 组件代码 Review
- 前端性能优化
- 前后端并行开发
- Mock 数据生成

**注意事项：**
- React Hook 要遵循 Hook 规则，避免在条件语句中使用
- 性能优化要基于实际测量，不要过度优化
- Mock 数据应覆盖各种边界情况，提高前端代码的健壮性
- 可访问性要从一开始就考虑，不要事后补救
- 状态管理要合理，避免 Prop Drilling 和过度全局化
