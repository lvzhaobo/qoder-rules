# AI Coding 深度提示词库

> 本文档提供面向企业级研发场景的系统化提示词模板,覆盖个人效能、架构分析、可视化、项目管理、研发运维、产品设计、前后端开发及测试运维等全栈领域。

**使用说明**:
- 文中 `[占位符]` 需替换为实际内容
- 每个提示词模板包含:目标定位、详细指令、预期输出及效果说明

---

## 1. 个人效能 (Personal Productivity)

### 1.1 遗留代码理解 (Legacy Code)

**适用角色**:全员通用

**目标**:快速接手看不懂的"祖传代码"

**提示词模板**:

```
我需要接手维护这段 [Java/Python] 遗留代码。请作为资深架构师:

1. **逻辑可视化**:用 Mermaid 流程图画出其核心执行路径。
2. **业务翻译**:用通俗的语言解释这段代码在做什么业务(注意:它似乎处理了 [订单状态流转])。
3. **风险预警**:指出代码中硬编码的"魔术数字"和潜在的 NullPointerException 风险。
4. **重构建议**:如果不破坏现有逻辑,如何用 [Java 8 Stream] 简化它?

代码片段:
`[粘贴代码]`
```

**预期输出**:
- ✅ 可视化的 Mermaid 流程图
- ✅ 通俗易懂的业务逻辑说明
- ✅ 防御性编程的改进建议

**效果**:降低新人上手老旧系统的门槛,减少因理解偏差导致的 Bug。

## 2. 深度分析 (Analysis)

### 2.1 全链路扫描 (Deep Scan)

**适用角色**:架构师

**目标**:扫描依赖、环境与技术栈

**提示词模板**:

```
请对当前项目/模块进行深度扫描:

1. **依赖分析**:扫描 [pom.xml / package.json],列出所有第三方库及其版本,标记出过时或高风险的依赖。
2. **环境依赖**:识别代码中对特定 OS、JDK 版本或环境变量的隐式依赖。
3. **类依赖图**:分析类 [OrderService] 的耦合度,列出它依赖了哪些内部模块(如 `UserModule`, `PaymentModule`)。
4. **技术栈快照**:总结当前使用的核心技术栈(如 Spring Boot 2.7, React 18, MyBatis-Plus)。
```

**预期输出**:
- ✅ 依赖健康度报告
- ✅ 环境约束清单
- ✅ 模块耦合度拓扑图

**效果**:快速掌握项目全貌,识别架构腐化风险。

---

### 2.2 性能与安全 (Performance & Security)

**适用角色**:安全专家

**目标**:发现性能瓶颈与安全漏洞

**提示词模板**:

```
请作为资深安全专家和性能调优专家分析此代码:

1. **性能瓶颈**:找出时间复杂度 > O(n) 的循环、未关闭的 I/O 流、大对象内存分配及 N+1 查询问题。
2. **安全漏洞**:扫描 OWASP Top 10 风险(特别是 SQL 注入、XSS、越权访问、敏感信息硬编码)。
3. **并发风险**:检查是否存在死锁、竞态条件或线程不安全的操作。
4. **修复方案**:针对发现的每个问题,提供具体的代码修复建议。
```

**预期输出**:
- ✅ 性能热点清单
- ✅ 安全漏洞审计报告
- ✅ 并发安全性评估

**效果**:在代码提交前消除"炸弹",提升系统稳健性。

### 2.3 红队挑战 (Red Teaming)

**适用角色**:反思者

**目标**:模拟高层质疑与极限边界挑战

**提示词模板**:

```
你是一位挑剔的"反思者" (Devil's Advocate)。请批判性地审视这段代码/方案:

1. **盲点挖掘**:我遗漏了哪些极端的边界情况?(如:并发量激增 100 倍、网络中断、数据库宕机)。
2. **漏洞反思**:如果黑客要攻击这个功能,他们会从哪里下手?
3. **高层拷问**:如果我把这个方案提交给 CTO,针对"扩展性"、"维护成本"和"ROI",他会问哪 3 个最尖锐的问题?请模拟他的语气提问。
4. **自我否定**:是否存在更简单、更低成本的替代方案?
```

**预期输出**:
- ✅ 未覆盖的边缘案例清单
- ✅ 攻击面分析
- ✅ 模拟 CTO 的尖锐提问 (Q&A)

**效果**:打破思维定势,提前准备高层汇报的防御性问答 (FAQ)。

### 3.2 复杂时序图 (Sequence)

**适用角色**:开发/架构

**目标**:梳理复杂的异步业务流程

**提示词模板**:

```
请生成 Mermaid 时序图,描述"订单支付成功后的回调处理"流程:

1. **参与方**:PaymentService, OrderService, MQ, InventoryService, NotificationService。
2. **流程**:
   - PaymentService 收到回调 -> 更新本地流水 -> 发送 MQ 消息。
   - OrderService 消费 MQ -> 幂等检查 -> 更新订单状态。
   - OrderService -> 调用 InventoryService 扣减库存 (RPC)。
   - 并行:调用 NotificationService 发送短信。
3. **异常**:如果库存扣减失败,需体现补偿逻辑 (Compensating Transaction)。
```

**预期输出**:
- ✅ 包含异步/并行逻辑的时序图
- ✅ 异常路径的可视化

**效果**:将晦涩的文字流程转化为直观的时序图,减少沟通歧义。

## 5. 运维部署 (DevOps)

### 5.1 阿里云 ECS 部署 (Cloud Deploy)

**适用角色**:运维工程师

**目标**:生成自动化部署脚本

**提示词模板**:

```
我需要将一个 Spring Boot JAR 包部署到阿里云 ECS (CentOS 7)。

请编写一个 Shell 脚本 (`deploy.sh`):
1. **环境检查**:检查是否安装 JDK 17,未安装则自动使用 Yum 安装 (Alibaba Dragonwell)。
2. **进程管理**:优雅停止旧的 Java 进程 (kill -15 -> wait -> kill -9)。
3. **备份**:将旧 JAR 包备份到 `backup/` 目录,带上时间戳。
4. **启动**:使用 `nohup` 启动新 JAR,配置 JVM 参数 (`-Xms512m -Xmx512m`),并将日志重定向到 `app.log`。
5. **健康检查**:脚本最后 curl 本地 `actuator/health` 接口验证启动是否成功。
```

**预期输出**:
- ✅ 健壮的 Shell 部署脚本
- ✅ 包含备份与健康检查机制

**效果**:标准化单机部署流程,减少人为操作失误。

**预期输出**:
- ✅ 完整的流水线 YAML 配置文件

**效果**:实现"提交即部署",保障代码交付流水线的标准化与自动化。

## 6. 管理视窗 (Management)

**适用角色**:研发总监

**目标**:评估系统的稳定性指标

**提示词模板**:

```
请作为 SRE 专家,根据以下 Prometheus/Grafana 监控数据快照(或日志片段),分析系统健康状况:

1. **黄金指标**:分析延迟 (Latency)、流量 (Traffic)、错误 (Errors)、饱和度 (Saturation)。
2. **异常模式**:是否存在周期性的 Full GC?是否存在连接池泄露迹象?
3. **SLO 风险**:当前错误率是否消耗了过多的错误预算 (Error Budget)?
4. **建议**:给出 3 条提升系统可用性的架构建议(如:熔断、降级、扩容)。

数据片段:`[粘贴监控数据]`
```

**预期输出**:
- ✅ 系统健康度诊断报告
- ✅ SRE 视角的优化建议

**效果**:量化评估系统稳定性,用数据指导架构优化方向。
